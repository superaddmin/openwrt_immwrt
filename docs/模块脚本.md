## network.sh

基础设施模块,被所有其他模块隐式依赖。提供统一的网络重试机制。

### network_retry

通用命令重试封装。

```bash
network_retry <command> [args...]
```

- 最大重试 `NETWORK_RETRY_MAX`(默认 5)
- 初始延迟 `NETWORK_RETRY_DELAY`(默认 5s),指数退避(×2)
- 命令成功立即返回,失败重试

### git_retry

git 命令专用重试,额外设置 `http.lowSpeedLimit=1000 http.lowSpeedTime=60`(1KB/s 持续 60s 视为失败)。

```bash
git_retry clone --depth 1 -b main <url> <dir>
git_retry -C <dir> sparse-checkout set <paths...>
git_retry ls-remote <url> <ref>
```

**clone 失败清理**:若 clone 目标已部分创建,失败时自动 `rm -rf` 清理,避免残留导致下次失败。

### curl_retry / wget_retry

```bash
curl_retry -fsSL <url> -o <file>
wget_retry -qO- <url>
```

底层调用 `network_retry`,curl/wget 自身也带 `--retry 3 --retry-delay 2`。

## general.sh

支持两种模式,由环境变量 `WRT_LOCAL_SOURCE` 控制(由 [build.sh](../build.sh) 在 submodule 模式下设为 `1`):

### clone_repo

```bash
clone_repo
```

- **`WRT_LOCAL_SOURCE=1`(submodule 模式,默认)**:校验 `BUILD_DIR`(此时为 `action_build`)存在即返回,**不执行 git clone**。源码已由 [prepare-source.sh](../scripts/prepare-source.sh) 同步好。
- **`WRT_LOCAL_SOURCE=0`(动态 clone 模式)**:若 `BUILD_DIR` 不存在,`git_retry clone --depth 1 -b "$REPO_BRANCH" "$REPO_URL" "$BUILD_DIR"`。失败则 exit 1。

### clean_up

```bash
clean_up
```

清理构建目录(`action_build/` 或动态 clone 目录)的临时状态:
- 删除 `.config`
- 删除 `tmp/` 目录
- 删除 `logs/*`(**已知 bug**:引号导致 glob 不展开,实际只删除字面 `logs/*`)
- `./scripts/feeds clean`
- 重建 `tmp/` 并写入 `tmp/.build` 标记

### reset_feeds_conf

```bash
reset_feeds_conf
```

- **`WRT_LOCAL_SOURCE=1`(submodule 模式,默认)**:跳过所有 git 操作,直接返回。副本已由 prepare-source.sh 从 submodule 同步,**不需要也无法 git reset/pull**(副本非 git 仓库)。
- **`WRT_LOCAL_SOURCE=0`(动态 clone 模式)**:在 BUILD_DIR 内执行 `git_retry reset --hard "origin/$REPO_BRANCH"` + `git_retry clean -f -d` + `git_retry pull`;若 `COMMIT_HASH != "none"`,checkout 到指定 commit。

## feeds.sh

### get_feeds_path

```bash
get_feeds_path
# 输出: $BUILD_DIR/feeds.conf.default 或 $BUILD_DIR/feeds.conf
```

优先用 `feeds.conf`,回退到 `feeds.conf.default`。

### append_feed_if_missing

```bash
append_feed_if_missing <feeds_path> <match_pattern> <feed_entry>
```

若 feeds 文件中不存在 `match_pattern`,追加 `feed_entry`。自动确保文件末尾有换行。

### update_feeds

```bash
update_feeds
```

定制 feeds 列表:
- 删除所有注释行
- 删除 `packages_ext` feed
- 删除已有的 `small8` / `custom_feed` feed
- 注入 `openwrt_bandix` 与 `luci_app_bandix` feed
- 创建空的 `include/bpf.mk`(若不存在)
- 执行 `./scripts/feeds update -a`

### install_feeds

```bash
install_feeds
```

- `./scripts/feeds update -i`(只更新 feeds 索引,不重新 clone)
- `./scripts/feeds install -a -f`(强制安装所有 feeds 到 package/feeds/)

## docker.sh

**最复杂的模块**。重写 dockerd / dockerman init 脚本,强制使用 nftables 后端(默认是 iptables)。

### 设计目标

OpenWRT fw4 默认使用 nftables,但 dockerd 默认用 iptables,导致 Docker 网络规则不生效。本模块通过原位补丁 dockerd init 脚本,使其:
1. 默认 `firewall_backend=nftables`
2. 用 nftables 创建 `docker-user` 表与 forward 链
3. 在 nftables 模式下跳过 iptables 链管理
4. 添加 IP 转发前置校验

### 主要函数

#### docker_stack_sync_nftables_compat

入口函数,协调 dockerd 与 dockerman 两端补丁。

```bash
docker_stack_sync_nftables_compat <build_dir> [dry_run=0] [storage_driver=]
```

流程:
1. 解析 dockerd Makefile / config / init / sysctl 路径
2. `_docker_stack_update_dockerd_depends_block`:重写 dockerd Makefile DEPENDS(切换到 nftables 兼容依赖集)
3. `_docker_stack_fix_dockerd_vendored_checks`:修复 vendored 版本校验(容忍缺失 installer 文件)
4. `_docker_stack_ensure_nftables_init_support`:补丁 dockerd init
5. `docker_stack_sync_dockerman_nftables_compat`:补丁 dockerman init
6. 设置 dockerd config `firewall_backend=nftables`
7. 修复 dockerd config 注释
8. 设置 sysctl `net.ipv4.ip_forward=1` 与 `net.ipv6.conf.all.forwarding=1`

#### _docker_stack_ensure_nftables_init_support

若 dockerd init 已支持 nftables(检测特定字符串),只调用 `_docker_stack_patch_iptables_dispatch`。否则依次:
- `_docker_stack_patch_nft_prereq_block`:注入 `verify_nftables_prerequisites` 等前置校验函数
- `_docker_stack_patch_process_config_nftables`:重写 `process_config` 函数,读取/迁移 `firewall_backend`
- `_docker_stack_patch_service_error_handling`:`process_config` 失败时 `return 1`
- `_docker_stack_patch_iptables_dispatch`:`iptables_add_blocking_rule` 增加 nftables 分支
- `_docker_stack_patch_append_nft_rule_helpers`:追加 `nftables_create_blocking_table` / `nftables_add_blocking_rules` 函数

#### docker_stack_sync_dockerman_nftables_compat

补丁 dockerman init:
- `_docker_stack_patch_dockerman_backend_helpers`:注入 `dockerman_firewall_backend` / `dockerman_use_iptables` 辅助函数
- `_docker_stack_patch_dockerman_start_service`:nftables 模式下跳过 DOCKER-MAN iptables 链管理

### 注意事项

此模块大量使用 awk 原位重写上游脚本,**依赖上游文件的具体格式**。上游 dockerd init 格式变更可能导致补丁静默失效(幂等检查只检测"是否已补丁",无法发现"目标行已变样")。这是本项目最大的维护风险点。

## 添加新定制步骤

### 1. 选择模块

| 类型 | 模块 |
|------|------|
| 网络相关 | network.sh(基础设施)/ feeds.sh |
| 包管理 | packages.sh |
| 系统配置/补丁 | system.sh |
| Docker 相关 | docker.sh |
| 编译修复 | system.sh |

### 2. 编写函数

遵循幂等模式:

```bash
my_custom_step() {
    local target_file="$BUILD_DIR/path/to/file"

    # 检查文件存在
    if [ ! -f "$target_file" ]; then
        echo "my_custom_step: $target_file not found, skip"
        return 0
    fi

    # 检查是否已应用
    if grep -q "my_marker" "$target_file"; then
        echo "my_custom_step: already applied, skip"
        return 0
    fi

    # 应用修改
    sed -i 's/old/new/g' "$target_file"
    echo "my_custom_step: applied"
}
```

### 3. 注册到 update.sh
