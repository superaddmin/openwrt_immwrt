# 构建流程

> 本文档逐步解析从执行 `./build.sh` 到产出固件的完整流程。

## 入口与参数解析

[build.sh](../build.sh) 是唯一入口,支持三种调用方式:

```bash
# 1. 完整命令行
./build.sh <device> [normal|debug|container|container_debug]

# 2. 交互式(无参数,在 TTY 下)
./build.sh
# → 列出支持的设备,输入编号选择
# → 列出 4 种编译模式,输入编号选择

# 3. 非交互式无参数(如 CI 重定向)
./build.sh < /dev/null
# → 打印用法与设备列表,exit 1
```

### 设备列表发现

[build.sh#collect_supported_devs](../build.sh#L24-L45) 动态扫描 `compilecfg/*.ini`,并与 `deconfig/<name>.config` 配对:

```bash
for ini_file in "$BASE_PATH"/compilecfg/*.ini; do
    dev_key=$(basename "$ini_file" .ini)
    if [[ -f "$BASE_PATH/deconfig/$dev_key.config" ]]; then
        SUPPORTED_DEVS+=("$dev_key")
    fi
done
```

只有 ini 与 config 同时存在才算有效设备。设备列表按字母序排序。

## 阶段一:配置读取

```bash
CONFIG_FILE="$BASE_PATH/deconfig/$Dev.config"
INI_FILE="$BASE_PATH/compilecfg/$Dev.ini"

REPO_URL=$(read_ini_by_key "REPO_URL")
REPO_BRANCH=$(read_ini_by_key "REPO_BRANCH")     # 默认 main
BUILD_DIR=$(read_ini_by_key "BUILD_DIR")
COMMIT_HASH=$(read_ini_by_key "COMMIT_HASH")     # 默认 none
```

ini 解析用 awk 实现([build.sh#L176-L179](../build.sh#L176-L179)),简单按 `=` 分割取值。

**CI 特殊处理**:若存在 `action_build/` 目录,强制 `BUILD_DIR="action_build"`([build.sh#L277-L279](../build.sh#L277-L279))。

## 阶段二:容器编译分支

若 `Build_Mod` 为 `container` 或 `container_debug`,走容器路径,不进入本地编译流程:

```bash
run_container_build() {
    build_target_sdk=$(read_ini_by_key "BUILD_TARGET_SDK")
    # 默认 immortalwrt/sdk:openwrt-25.12

    prepare_container_image "$build_target_sdk" "$container_name"
    # → docker pull SDK 镜像
    # → 生成临时 Dockerfile(安装 sudo/git/jq/build-essential 等)
    # → docker build 构建编译镜像

    docker run --rm -it \
        -v "$REPO_ROOT":/build \
        --shm-size=8g --ipc=shareable \
        --ulimit nofile=65535:65535 \
        "$container_name" \
        bash wrt_core/build_container.sh "$Dev" "$container_build_mod"
}
```

容器内执行 [build_container.sh](../wrt_core/build_container.sh):
- `container` 模式:直接 `./build.sh "$Dev"` 完整编译
- `container_debug` 模式:编译到 defconfig 后 `exec bash -i` 进入交互 shell

## 阶段三:源码定制(update.sh)

## 阶段四:配置应用

回到 [build.sh#apply_config](../build.sh#L255-L268):

```bash
apply_config() {
    \cp -f "$CONFIG_FILE" "$BASE_PATH/../$BUILD_DIR/.config"

    # IPQ 设备追加 NSS 配置
    if grep -qE "(ipq60xx|ipq807x)" ".../.config" && \
       ! grep -q "CONFIG_GIT_MIRROR" ".../.config"; then
        cat "$BASE_PATH/deconfig/nss.config" >> ".../.config"
    fi

    cat "$BASE_PATH/deconfig/compile_base.config" >> ".../.config"
    cat "$BASE_PATH/deconfig/docker_deps.config" >> ".../.config"
    cat "$BASE_PATH/deconfig/proxy.config" >> ".../.config"
}
```

**配置合并语义**:多个 config 用 `cat >>` 追加,后追加的覆盖先追加的同名选项。最后 `make defconfig` 按 OpenWRT Kconfig 规则展开依赖、解决冲突(last-write-wins)。

### remove_uhttpd_dependency

[build.sh#L233-L243](../build.sh#L233-L243) 特殊处理:若启用 `luci-app-quickfile`(nginx 后端),从 luci Makefile 移除 `luci-light`(uhttpd)依赖,避免 Web 服务器冲突。

## 阶段五:defconfig 与 x64 修正

```bash
cd "$BUILD_DIR"
make defconfig
```

`make defconfig` 根据现有 .config 自动:
- 展开包依赖(选 A 自动选 B)
- 设置未指定项为默认值
- 解决冲突

**x64 特殊修正**([build.sh#L289-L294](../build.sh#L289-L294)):distfeeds 配置中架构硬编码为 `aarch64_cortex-a53`,x64 设备需 sed 改为 `x86_64`。

## 阶段六:debug 模式退出

```bash
if [[ $Build_Mod == "debug" ]]; then
    exit 0
fi
```

`debug` 模式到此停止,便于检查 `.config` 内容、feeds 状态等。

## 阶段七:编译

```bash
# 清理旧产物
find "$TARGET_DIR" -type f \( -name "*.bin" -o -name "*.manifest" ... \) -exec rm -f {} +

# 下载源码包(带重试)
make download -j$(($(nproc) * 2))

# 编译,失败则单线程详细日志
make -j$(($(nproc) + 1)) || make -j1 V=s
```

- `make download` 用 2 倍 CPU 核数并发下载
- `make` 用 nproc+1 并发编译
- 失败时 `make -j1 V=s` 单线程输出详细日志,便于定位

## 阶段八:产物收集

```bash
FIRMWARE_DIR="$BASE_PATH/../firmware"
\rm -rf "$FIRMWARE_DIR"
mkdir -p "$FIRMWARE_DIR"

find "$TARGET_DIR" -type f \( \
    -name "*.bin" -o -name "*.manifest" -o -name "*efi.img.gz" \
    -o -name "*.itb" -o -name "*.fip" -o -name "*.ubi" \
    -o -name "*rootfs.tar.gz" \
\) -exec cp -f {} "$FIRMWARE_DIR/" \;
```

收集的产物类型:
- `*.bin` — 常规固件镜像
- `*.itb` — U-Boot Image Tree Blob(ARM 设备)
- `*.fip` — FIT Image(IPQ 设备)
- `*.ubi` — UBI 镜像
- `*efi.img.gz` — x86 EFI 镜像
- `*rootfs.tar.gz` — 根文件系统打包

**CI 额外清理**:若 `action_build/` 存在,执行 `make clean` 释放空间。

## CI 特有流程

[.github/workflows/build_wrt.yml](../.github/workflows/build_wrt.yml) 在本地流程基础上增加:

### 缓存

```yaml
- name: Cache Dependencies
  uses: actions/cache@v6.1.0
  with:
    path: |
      ./action_build/.ccache
      ./action_build/staging_dir
    key: ${{ matrix.os }}-${{ hashFiles('**/repo_flag') }}-${{ env.BUILD_DATE }}
    restore-keys: |
      ${{ matrix.os }}-${{ hashFiles('**/repo_flag') }}-
```

- 缓存 key 含**日期**,每日新缓存;restore-keys 前缀匹配回退到最近缓存
- `repo_flag` 文件由 [pre_clone_action.sh](../wrt_core/pre_clone_action.sh) 写入,内容为 `REPO_URL/REPO_BRANCH`,用于检测上游变更

### 缓存刷新

```yaml
- name: Refresh the cache
  run: |
    find "./action_build/staging_dir" -type d -name "stamp" -not -path "*target*" | while read -r dir; do
        find "$dir" -type f -exec touch {} +
    done
```

touch stamp 文件防止 make 因时间戳误判需要重新编译。

### 旧缓存清理

```yaml
- name: Delete Old Cache
  run: |
    gh cache list --key ${{ matrix.os }}-${{ hashFiles('**/repo_flag') }}- --json key --jq '.[] | .key' | while read -r key; do
      gh cache delete "$key"
    done
```

删除同前缀的所有旧缓存,避免缓存膨胀。

### 镜像源清理

[pre_clone_action.sh#L47-L51](../wrt_core/pre_clone_action.sh#L47-L51) 在 CI 中移除国内镜像源,避免 GitHub Actions 在境外网络访问 .cn 镜像失败:

```bash
sed -i '/.cn\//d; /tencent/d; /aliyun/d' "$PROJECT_MIRRORS_FILE"
```

## 故障排查

### 编译失败

1. 查看完整日志:CI 中下载 artifact,本地看终端输出
2. 若 `make -j` 失败,自动 fallback 到 `make -j1 V=s`,看详细错误
3. 常见原因:网络下载失败(检查 retry 日志)、补丁失效(检查上游是否改格式)、依赖缺失(检查 feeds install)

### 配置不生效

1. 用 `debug` 模式停在 defconfig 后,检查 `.config`:
   ```bash
   ./build.sh <device> debug
   grep "CONFIG_XXX" <BUILD_DIR>/.config
   ```
2. 确认配置合并顺序:设备 config → nss → compile_base → docker_deps → proxy,后者覆盖前者

### 容器编译失败

1. 检查 Docker 是否运行
2. 检查 SDK 镜像是否可拉取
3. 用 `container_debug` 模式进入容器内手动排查
