# 构建流程

> 本文档逐步解析从执行 `./build.sh` 到产出固件的完整流程，并补充独立 `.ipk` 包构建链路。

## 入口与参数解析

[build.sh](../build.sh) 是唯一入口,支持三种调用方式:

```bash
# 1. 完整命令行
./build.sh <device> [normal|debug|container|container_debug]

# 2. 交互式(无参数,在 TTY 下)
./build.sh
# → 列出支持的设备,输入编号选择
# → 列出 4 种编译模式,输入编号选择

# 3. 非交互式无参数(如 CI 重定向)
./build.sh < /dev/null
# → 打印用法与设备列表,exit 1
```

### 设备列表发现

[build.sh#collect_supported_devs](../build.sh#L24-L45) 动态扫描 `compilecfg/*.ini`,并与 `deconfig/<name>.config` 配对。只有 ini 与 config 同时存在才算有效设备。

## 阶段一:配置读取

```bash
CONFIG_FILE="$BASE_PATH/deconfig/$Dev.config"
INI_FILE="$BASE_PATH/compilecfg/$Dev.ini"

REPO_URL=$(read_ini_by_key "REPO_URL")
REPO_BRANCH=$(read_ini_by_key "REPO_BRANCH")     # 默认 main
BUILD_DIR=$(read_ini_by_key "BUILD_DIR")         # 此处即 source-id
COMMIT_HASH=$(read_ini_by_key "COMMIT_HASH")     # 默认 none
```

**关键**:`BUILD_DIR` 在 submodule 模式下就是 source-id(如 `imm-nss`),对应 `sources/<BUILD_DIR>`。

ini 解析用 awk 实现([build.sh#L176-L179](../build.sh#L176-L179))。

## 阶段二:容器编译分支

若 `Build_Mod` 为 `container` 或 `container_debug`,走容器路径,不进入本地编译流程:

```bash
run_container_build() {
    build_target_sdk=$(read_ini_by_key "BUILD_TARGET_SDK")
    # 默认 immortalwrt/sdk:openwrt-25.12
    prepare_container_image "$build_target_sdk" "$container_name"
    # → docker pull SDK 镜像,生成 Dockerfile 安装编译依赖,docker build
    docker run --rm -it -v "$REPO_ROOT":/build ...
        bash wrt_core/build_container.sh "$Dev" "$container_build_mod"
}
```

容器内执行 [build_container.sh](../wrt_core/build_container.sh):
- `container` 模式:直接 `./build.sh "$Dev"` 完整编译
- `container_debug` 模式:编译到 defconfig 后 `exec bash -i` 进入交互 shell

## 阶段三:源码准备(submodule 模式,默认)

这是新架构的核心阶段。[build.sh#L278-L284](../build.sh#L278-L284):

```bash
if [[ ${WRT_USE_DYNAMIC_CLONE:-0} == "1" ]]; then
    echo "Using dynamic clone mode for $Dev"
    # 旧模式:不调用 prepare-source,BUILD_DIR 保持 ini 值
else
    "$REPO_ROOT/scripts/prepare-source.sh" "$Dev"
    BUILD_DIR="action_build"
    export WRT_LOCAL_SOURCE=1
fi
```

### prepare-source.sh 工作流程

[scripts/prepare-source.sh](../scripts/prepare-source.sh):

1. **解析设备源码路径**:`resolve_device_source "$device"` → 读 ini 的 `BUILD_DIR` → 返回 `sources/<source-id>`
2. **校验 submodule 已初始化**:检查 `sources/<source-id>/.git` 存在,否则提示运行 `git submodule update --init --depth 1 -- sources/<source-id>` 并 exit 1
3. **校验构建路径**:断言 `build_copy_path == $repo_root/action_build`,防止误操作其他目录
4. **同步源码**:
   - 优先 `rsync -a --delete`(带排除项)
   - 无 rsync 时回退 `tar` 管道,且**保留 .ccache 与 staging_dir**(mv 到临时目录,重建后 mv 回)
5. **写入源码锁**:`write_source_lock` → `action_build/.source-lock`(记录 device/source_path/source_url/source_branch/source_commit/locked_at_utc)

### 同步排除项

[source-common.sh#source_sync_excludes](../scripts/source-common.sh):

```
.git
.ccache
bin
build_dir
dl
logs
staging_dir
tmp
feeds
package/feeds
.config
.config.old
```

这些是构建产物或缓存,不从 submodule 同步,保证副本干净。

### 旧模式(动态 clone)

设置 `WRT_USE_DYNAMIC_CLONE=1` 时跳过 prepare-source,`BUILD_DIR` 保持 ini 值(如 `imm-nss`),后续 `clone_repo` 会动态 clone 到该目录。此模式仅用于兼容/调试。

## 阶段四:源码定制(update.sh)

调用 [update.sh](../wrt_core/update.sh),执行约 60 个定制步骤。完整列表见 [模块脚本文档](模块脚本.md)。

**submodule 模式的特殊行为**:由于 `WRT_LOCAL_SOURCE=1` 已导出:
- [clone_repo](../wrt_core/modules/general.sh#L4-L21):检测到 local source 模式,只校验 `BUILD_DIR`(此时为 `action_build`)存在即返回,**不执行 git clone**
- [reset_feeds_conf](../wrt_core/modules/general.sh#L45-L57):检测到 local source 模式,跳过 `git reset/pull/checkout`,**因为副本已由 prepare-source.sh 准备好**

其余步骤(clean_up、update_feeds、所有 fix_/update_/set_/change_ 函数)行为不变,在 `action_build/` 上原地修改。

按职责分组的完整步骤见 [模块脚本文档](模块脚本.md),主要涵盖:

- **包管理**:移除重复包、sparse clone 第三方 feed
- **编译修复**:Linux 6.12/6.18 netfilter、Rust、Kconfig 递归依赖等
- **系统定制**:Argon 主题、LAN 地址、CPU 采样、构建签名
- **Docker nftables**:重写 dockerd/dockerman init
- **第三方服务**:SmartDNS / mosDNS / AdGuardHome / Lucky / EasyTier / OAF

## 阶段五:配置应用

回到 [build.sh#apply_config](../build.sh#L255-L268):

```bash
apply_config() {
    \cp -f "$CONFIG_FILE" "$BASE_PATH/../$BUILD_DIR/.config"
    # IPQ 设备追加 NSS 配置
    if grep -qE "(ipq60xx|ipq807x)" ".../.config" && ! grep -q "CONFIG_GIT_MIRROR"...; then
        cat "$BASE_PATH/deconfig/nss.config" >> ".../.config"
    fi
    cat "$BASE_PATH/deconfig/compile_base.config" >> ".../.config"
    cat "$BASE_PATH/deconfig/docker_deps.config" >> ".../.config"
    cat "$BASE_PATH/deconfig/proxy.config" >> ".../.config"
}
```

**配置合并语义**:多个 config 用 `cat >>` 追加,后追加的覆盖先追加的同名选项。最后 `make defconfig` 按 Kconfig 规则展开依赖、解决冲突(last-write-wins)。

### remove_uhttpd_dependency

若启用 `luci-app-quickfile`(nginx 后端),从 luci Makefile 移除 `luci-light`(uhttpd)依赖,避免 Web 服务器冲突。

## 阶段六:defconfig 与 x64 修正

```bash
cd "$BUILD_DIR"  # 即 action_build
make defconfig
```

`make defconfig` 自动展开包依赖、设置未指定项默认值、解决冲突。

**x64 特殊修正**([build.sh#L294-L299](../build.sh#L294-L299)):distfeeds 配置中架构硬编码为 `aarch64_cortex-a53`,x64 设备需 sed 改为 `x86_64`。

## 阶段七:debug 模式退出

```bash
if [[ $Build_Mod == "debug" ]]; then
    exit 0
fi
```

`debug` 模式到此停止,便于检查 `.config`、feeds 状态等。

## 阶段八:编译

```bash
# 清理旧产物
find "$TARGET_DIR" -type f \( -name "*.bin" -o -name "*.manifest" ... \) -exec rm -f {} +

# 下载源码包(带重试)
make download -j$(($(nproc) * 2))

# 编译,失败则单线程详细日志
make -j$(($(nproc) + 1)) || make -j1 V=s
```

- `make download` 用 2 倍 CPU 核数并发
- `make` 用 nproc+1 并发
- 失败时 `make -j1 V=s` 单线程详细日志定位

## 阶段九:产物收集

```bash
FIRMWARE_DIR="$BASE_PATH/../firmware"
\rm -rf "$FIRMWARE_DIR"
mkdir -p "$FIRMWARE_DIR"

find "$TARGET_DIR" -type f \( \
    -name "*.bin" -o -name "*.manifest" -o -name "*efi.img.gz" \
    -o -name "*.itb" -o -name "*.fip" -o -name "*.ubi" \
    -o -name "*rootfs.tar.gz" \
\) -exec cp -f {} "$FIRMWARE_DIR/" \;
```

产物类型:`*.bin`(常规镜像)、`*.itb`(U-Boot Image Tree Blob)、`*.fip`(FIT Image)、`*.ubi`(UBI 镜像)、`*efi.img.gz`(x86 EFI)、`*rootfs.tar.gz`(根文件系统)。

**空间回收**:若存在 `action_build/`,编译后执行 `make clean` 释放空间。

## CI 特有流程

[.github/workflows/build_wrt.yml](../.github/workflows/build_wrt.yml) 在本地流程基础上增加:

### 1. 按需初始化 submodule

```yaml
- name: Checkout
  uses: actions/checkout@v7.0.0
  with:
    submodules: false       # 不自动拉所有 submodule
    fetch-depth: 1

- name: Resolve Source
  run: |
    source_id=$(awk -F"=" '$1 == "BUILD_DIR" {print $2; exit}' "./wrt_core/compilecfg/${{ inputs.model }}.ini")
    echo "SOURCE_ID=$source_id" >> "$GITHUB_ENV"

- name: Init Target Source Submodule
  run: |
    git submodule sync -- "sources/${SOURCE_ID}"
    git submodule update --init --depth 1 -- "sources/${SOURCE_ID}"
```

CI 只初始化目标设备的**单个** submodule(`--depth 1`),避免拉取全部 6 个源码。

### 2. 源码校验

```yaml
- name: Validate Source
  run: |
    ./scripts/validate-source.sh "${{ inputs.model }}"
```

[validate-source.sh](../scripts/validate-source.sh) 校验:
- submodule 已在 `.gitmodules` 注册且已初始化
- 扫描 `wrt_core/` 下所有 `https://` URL,逐一校验是否在 `metadata/external-dependencies.tsv` 登记
- 缺失即 fail-fast

### 3. 缓存(基于源码 commit)

```yaml
- name: Export Source Commit
  run: |
    source_commit=$(git -C "sources/${SOURCE_ID}" rev-parse HEAD)

- name: Cache Dependencies
  uses: actions/cache@v6.1.0
  with:
    path: |
      ./action_build/.ccache
      ./action_build/staging_dir
    key: ${{ matrix.os }}-${{ inputs.model }}-${{ env.SOURCE_COMMIT }}-${{ hashFiles('wrt_core/**/*.sh', 'wrt_core/compilecfg/*.ini', 'wrt_core/deconfig/*.config', 'metadata/external-dependencies.tsv', 'scripts/*.sh') }}-${{ env.BUILD_DATE }}
    restore-keys: |
      ${{ matrix.os }}-${{ inputs.model }}-${{ env.SOURCE_COMMIT }}-
```

- 缓存 key 含 **source_commit**(精确到上游 commit)+ 脚本/配置/metadata 哈希 + 日期
- 源码 commit 变更或脚本变更即生成新缓存
- restore-keys 前缀匹配回退到同 commit 的最近缓存

### 4. 缓存刷新

```yaml
- name: Refresh the cache
  run: |
    find "./action_build/staging_dir" -type d -name "stamp" -not -path "*target*" | while read -r dir; do
        find "$dir" -type f -exec touch {} +
    done
```

touch stamp 文件防止 make 因时间戳误判需要重新编译。

### 5. 产物校验

```yaml
- name: Validate Firmware Output
  run: |
    test -d ./firmware
    count=$(find ./firmware -type f \( -name "*.manifest" -o -name "*.bin" ... \) | wc -l)
    test "$count" -gt 0
```

CI 额外校验产物数量 > 0,避免空产物误判成功。

### 6. 旧缓存清理

删除同前缀的所有旧缓存,避免缓存膨胀。

## 独立 IPK 构建链路

当只需要发布单个软件包时，使用 [scripts/build-ipk.sh](../scripts/build-ipk.sh)：

```bash
./scripts/build-ipk.sh x64_immwrt luci-app-timecontrol
./scripts/build-ipk.sh jdcloud_ipq60xx_immwrt feeds/custom_feed/lucky
```

### 阶段一:准备 action_build

脚本固定先调用：

```bash
./build.sh <device> debug
```

这一步只准备 `action_build/` 和 `make defconfig`，不进入整机固件编译。

### 阶段二:包目标规范化

脚本支持三种目标写法：

```text
luci-app-timecontrol
package/luci-app-timecontrol
feeds/custom_feed/lucky
```

内部统一转成 `package/...` 目标，再调用 `make <target>/download` 与 `make <target>/compile`。

### 阶段三:产物收集

脚本从 `action_build/bin/` 中收集 `.ipk`，输出到：

```text
ipk_artifacts/<device>/
```

默认收集“本次编译后新增或新更新”的 `.ipk`，也支持通过 `WRT_IPK_ARTIFACT_PATTERNS` 只匹配指定包名。

同时生成：

- `sha256sums.txt`
- `package-targets.txt`
- `artifact-patterns.txt`
- `source-lock.txt`

## IPK 发布工作流

新增 [release_ipk.yml](../.github/workflows/release_ipk.yml)：

- 触发方式：`workflow_dispatch`
- 输入：`model`、`package_targets`、`artifact_patterns`
- 流程：checkout → resolve source → targeted submodule init → validate source → restore cache → `scripts/build-ipk.sh` → validate ipk output → release

## 故障排查

### 编译失败

1. 查看完整日志:CI 中下载 artifact,本地看终端输出
2. 若 `make -j` 失败,自动 fallback 到 `make -j1 V=s`,看详细错误
3. 常见原因:网络下载失败(检查 retry 日志)、补丁失效(检查上游是否改格式)、依赖缺失(检查 feeds install)

### 配置不生效

1. 用 `debug` 模式停在 defconfig 后,检查 `.config`:
   ```bash
   ./build.sh <device> debug
   grep "CONFIG_XXX" action_build/.config
   ```
2. 确认配置合并顺序:设备 config → nss → compile_base → docker_deps → proxy,后者覆盖前者

### submodule 未初始化

```
Source submodule is not initialized: sources/imm-nss
Run: git submodule update --init --depth 1 -- sources/imm-nss
```

按提示执行对应命令。CI 会自动初始化;本地需手动。

### 外部依赖未登记

```
Unregistered external dependencies:
  https://github.com/xxx/yyy.git
```

在 [metadata/external-dependencies.tsv](../metadata/external-dependencies.tsv) 添加一行,字段参照已有条目。

### 容器编译失败

1. 检查 Docker 是否运行
2. 检查 SDK 镜像是否可拉取
3. 用 `container_debug` 模式进入容器内手动排查

### 独立 IPK 构建失败

1. 检查 `package_targets` 是否为当前 source tree 中真实存在的 OpenWrt 包目标
2. 先本地执行 `./build.sh <device> debug`，确认 `action_build/.config` 与 feeds 已准备完成
3. 若 `.ipk` 收集为空，补充 `artifact_patterns` 精确匹配主包与语言包
