# 技术架构

> 本文档阐述 OpenWRT ImmWRT 定制固件编译框架的分层架构、设计理念与数据流。

## 设计理念

### 核心问题

传统 OpenWRT 定制固件项目通常采用 **fork 上游仓库 + 直接修改** 的方式,存在三大痛点:

1. **同步困难**:上游更新后,merge 冲突频繁,fork 仓库容易落后
2. **维护成本高**:每个定制点都散落在源码各处,难以追踪"我改了什么"
3. **多设备复用难**:不同设备需要不同上游(IPQ 用 VIKINGYFY fork、MT7986 用 padavanonly fork、LibWrt 用 LiBwrt),fork 模式难以统一管理

### 本框架的解法

采用 **源码仓库 + 定制脚本 + 配置/补丁** 三分离架构:

- **源码仓库**:不在本仓库内,构建时按设备 ini 配置动态 clone 上游
- **定制脚本**:模块化的 shell 函数,每个函数做一件具体的事(打补丁、改包、修配置)
- **配置/补丁**:作为静态资源存放在 `wrt_core/patches/`,与脚本分离

构建时,脚本以**只读上游 + 原地修改本地副本**的方式工作,上游仓库保持纯净,所有定制记录在本仓库中,可追踪、可回滚、可复用。

## 关键设计模式

### 1. 编排与实现分离

[update.sh](../wrt_core/update.sh) 的 `main()` 函数是纯编排:

```bash
main() {
    clone_repo
    clean_up
    reset_feeds_conf
    update_feeds
    remove_unwanted_packages
    # ...约 60 个步骤
}
```

每个步骤是一个**独立函数**,定义在 `modules/*.sh` 中。修改定制流程只需调整 `main()` 中的调用顺序或增删条目,无需改动函数实现。

### 2. 网络重试封装

[network.sh](../wrt_core/modules/network.sh) 提供 `git_retry` / `curl_retry` / `wget_retry`,统一处理网络不稳定:

- 指数退避:5s → 10s → 20s → 40s → 80s
- 最大重试 5 次
- `git_retry clone` 失败时自动清理半成品目录,避免残留导致下次失败
- 通过环境变量 `NETWORK_RETRY_MAX` / `NETWORK_RETRY_DELAY` 可调

### 3. 幂等性

多数定制函数遵循"先检查再改"的幂等模式:

```bash
# docker.sh 中的典型模式
_docker_stack_patch_dockerman_backend_helpers() {
    local dockerman_init="$1"
    # 已打过补丁则直接返回
    grep -Fq 'dockerman_use_iptables() {' "$dockerman_init" && return 0
    # 否则打补丁
    ...
}
```

这使得**重复运行 update.sh 安全**,便于调试(`debug` 模式可重复执行)。

### 4. Sparse Checkout 优化

[install_custom_feed](../wrt_core/modules/packages.sh) 从 4 个第三方仓库拉取指定包时,使用 git sparse-checkout **只下载需要的目录**:

```bash
clone_args=(clone --depth 1 --filter=blob:none --sparse)
git_retry "${clone_args[@]}" "$repo_url" "$tmp_dir"
git_retry -C "$tmp_dir" sparse-checkout set "${packages[@]}"
```

相比完整 clone,大幅减少网络流量与磁盘占用。

### 5. Fail-Fast 校验

关键路径上设置校验闸门,缺失依赖立即报错而非让 make 失败:

```bash
# verify_custom_feed_installed_paths 检查 feeds 安装结果
if [ ${#missing_package_dirs[@]} -ne 0 ]; then
    printf '错误:...缺少以下仓库依赖路径:\n' >&2
    return 1
fi
```

### 6. 错误屏障

[update.sh](../wrt_core/update.sh) 顶部设置:

```bash
set -e
set -o errexit
set -o errtrace
trap 'error_handler' ERR
```

任何步骤失败立即停止,避免错误传播导致后续步骤在错误状态下运行。

## 数据流

### 完整构建数据流

```
用户执行 ./build.sh jdcloud_ipq60xx_immwrt
        │
        ▼
build.sh 读取 compilecfg/jdcloud_ipq60xx_immwrt.ini
        │  → REPO_URL=https://github.com/VIKINGYFY/immortalwrt.git
        │  → REPO_BRANCH=main
        │  → BUILD_DIR=imm-nss
        │
        ▼
update.sh 执行(对 imm-nss/ 目录)
        │
        ├─ clone_repo         → git clone VIKINGYFY/immortalwrt → imm-nss/
        ├─ clean_up            → 清理 .config / tmp / feeds
        ├─ reset_feeds_conf    → git reset --hard origin/main
        ├─ update_feeds        → 注入 bandix feed,./scripts/feeds update -a
        ├─ remove_unwanted_packages → 删除 passwall/ssr-plus 等重复包
        ├─ install_custom_feed → sparse clone 4 个第三方仓库到 custom_feed/
        ├─ fix_netfilter_kmod_clash → 修补 netfilter.mk (Linux 6.12 兼容)
        ├─ docker_stack_sync_nftables_compat → 重写 dockerd init
        ├─ ...(共约 60 步)
        └─ install_feeds       → ./scripts/feeds install -a -f
        │
        ▼
build.sh apply_config
        │  cp deconfig/jdcloud_ipq60xx_immwrt.config → imm-nss/.config
        │  cat nss.config >> .config (IPQ 设备)
        │  cat compile_base.config >> .config
        │  cat docker_deps.config >> .config
        │  cat proxy.config >> .config
        │
        ▼
cd imm-nss && make defconfig  (展开依赖、解决冲突)
        │
        ▼
make download -j24  (下载源码包,带重试)
        │
        ▼
make -j13  (编译,失败则 make -j1 V=s 详细日志)
        │
        ▼
find bin/targets -name "*.bin|*.itb|*.fip|..." → cp firmware/
        │
        ▼
输出 firmware/ 目录(含 .bin / .manifest / .itb 等)
```

### CI 构建数据流

GitHub Actions 流程略有不同:

```
workflow_dispatch(选设备) → pre_clone_action.sh(预克隆到 action_build/)
   → 缓存恢复(.ccache + staging_dir)→ build.sh → 上传 artifact
```

关键差异:
- CI 用 `action_build/` 替代 ini 中的 `BUILD_DIR`([build.sh#L277-279](../build.sh#L277-L279))
- 启用 ccache 加速(`CONFIG_CCACHE=y`)
- 缓存 key 基于设备 + 日期,每日构建新缓存

## 模块依赖关系

```
update.sh
  ├── source general.sh    → clone_repo, clean_up, reset_feeds_conf
  ├── source network.sh    → git_retry, curl_retry, wget_retry (被其他模块依赖)
  ├── source feeds.sh      → update_feeds, install_feeds, get_feeds_path
  ├── source packages.sh   → remove_unwanted_packages, install_custom_feed, update_*
  ├── source system.sh     → fix_*, update_*, set_*, change_*
  ├── source cups.sh       → fix_cups_libcups_avahi_depends
  └── source docker.sh     → docker_stack_sync_nftables_compat

network.sh 是基础设施,被所有其他模块隐式依赖(git_retry 等)
packages.sh 中的 get_custom_feed_* 系列函数被 system.sh / cups.sh 调用
```

## 扩展点

### 添加新设备

1. 创建 `compilecfg/<new_device>.ini`(指定上游仓库)
2. 创建 `deconfig/<new_device>.config`(指定平台与包)
3. 若是新平台,可能需要在 system.sh 增加平台专属补丁
4. 在 `.github/workflows/build_wrt.yml` 的 `options` 添加设备名

### 添加新定制步骤

1. 在合适的 module(如 `system.sh`)中编写函数,遵循幂等模式
2. 在 `update.sh` 的 `main()` 中按依赖顺序插入调用
3. 若需要补丁文件,放入 `patches/`

### 添加新第三方包源

修改 [packages.sh#install_custom_feed](../wrt_core/modules/packages.sh) 中的 `custom_feed_sources` 数组:

```bash
custom_feed_sources=(
    "kenzok8/small-package|https://github.com/kenzok8/small-package.git||${base_custom_feed_packages[*]}"
    "sbwml/luci-app-mosdns|https://github.com/sbwml/luci-app-mosdns.git|v5|mosdns luci-app-mosdns"
    # 新增:
    "your/repo|https://github.com/your/repo.git|main|pkg-a pkg-b"
)
```

格式:`repo_label|repo_url|repo_branch|pkg1 pkg2 pkg3`

## 架构权衡

### 优点
- **上游纯净**:本仓库不含 OpenWRT 源码,体积小,clone 快
- **定制可追踪**:所有修改以脚本形式存在,git 可追溯
- **多上游支持**:同一套定制脚本适配 4 个不同上游分支
- **CI 友好**:无源码则无需处理源码 merge 冲突

### 代价
- **脆弱性**:定制脚本依赖上游文件的具体格式(如 docker.sh 的 awk 匹配),上游改格式则补丁失效
- **调试复杂度**:构建时才 clone + 修改,出问题需在 BUILD_DIR 内排查
- **无单元测试**:shell 脚本缺少测试覆盖,依赖实际编译验证
