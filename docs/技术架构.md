# 技术架构

> 本文档阐述 OpenWRT ImmWRT 定制固件编译框架的分层架构、设计理念与数据流。

## 设计理念

### 核心问题

传统 OpenWRT 定制固件项目通常采用 **fork 上游仓库 + 直接修改** 的方式,存在三大痛点:

1. **同步困难**:上游更新后,merge 冲突频繁,fork 仓库容易落后
2. **维护成本高**:每个定制点都散落在源码各处,难以追踪"我改了什么"
3. **多设备复用难**:不同设备需要不同上游(IPQ 用 VIKINGYFY fork、MT7986 用 padavanonly fork、LibWrt 用 LiBwrt),fork 模式难以统一管理

### 本框架的解法

早期版本采用"构建时动态 clone 上游"的方式。现已演进为 **上游源码 submodule + 定制脚本 + 配置/补丁** 三分离架构:

- **上游源码**:以 git submodule 形式纳入 `sources/<source-id>/`,构建时只初始化目标设备需要的那一个 submodule,再同步到构建副本 `action_build/`
- **定制脚本**:模块化的 shell 函数,每个函数做一件具体的事(打补丁、改包、修配置)
- **配置/补丁**:作为静态资源存放在 `wrt_core/patches/`,与脚本分离
- **发布链路分层**:整机固件继续输出到 `firmware/`,独立软件包输出到 `ipk_artifacts/`

这样带来三个好处:
- **源码可追踪**:上游 commit 由 submodule 指针锁定,变更走 git 流程可审计
- **CI 稳定**:不再依赖运行时 clone,源码版本在仓库中固定
- **定制记录完整**:所有修改以脚本形式存在,git 可追溯;外部依赖 URL 集中登记在 `metadata/external-dependencies.tsv`

### 旧流程的兼容

通过环境变量 `WRT_USE_DYNAMIC_CLONE=1` 可回退到旧的动态 clone 流程,便于临时测试或对照:

```bash
WRT_USE_DYNAMIC_CLONE=1 ./build.sh x64_immwrt debug
```

## 关键设计模式

### 1. 编排与实现分离

[update.sh](../wrt_core/update.sh) 的 `main()` 函数是纯编排:

```bash
main() {
    clone_repo
    clean_up
    reset_feeds_conf
    update_feeds
    remove_unwanted_packages
    # ...约 60 个步骤
}
```

每个步骤是一个**独立函数**,定义在 `modules/*.sh` 中。修改定制流程只需调整 `main()` 中的调用顺序或增删条目,无需改动函数实现。

### 2. 构建副本机制(核心创新)

构建不直接在 `sources/<source-id>/` submodule 工作树上进行,而是先同步到 `action_build/` 副本:

```
sources/imm-nss/  (submodule,只读,保护源码指针)
       │
       │ prepare-source.sh (rsync/tar,排除 .git/.ccache/staging_dir/bin/dl/...)
       ▼
action_build/  (构建副本,所有定制与编译在此进行)
```

**为什么需要副本**:
- **保护 submodule**:submodule 工作树保持干净,定制脚本的修改不污染源码指针
- **缓存隔离**:`action_build/.ccache` 与 `action_build/staging_dir` 可独立缓存,重建副本时保留
- **可重复**:`action_build/` 是 gitignore 的临时产物,删除重建不影响仓库

**同步排除项**([source-common.sh#source_sync_excludes](../scripts/source-common.sh)):`.git`、`.ccache`、`bin`、`build_dir`、`dl`、`logs`、`staging_dir`、`tmp`、`feeds`、`package/feeds`、`.config`、`.config.old`

**两种同步实现**([prepare-source.sh](../scripts/prepare-source.sh)):
- 优先 `rsync -a --delete`(带排除项),高效增量同步
- 无 rsync 时回退 `tar` 管道,且**保留 .ccache 与 staging_dir**(先 mv 到临时目录,重建后再 mv 回)

### 2.5 独立包发布链路

在整机固件链路之外，仓库新增了“独立 `.ipk` 包构建”路径：

```
./scripts/build-ipk.sh <device> <package-target...>
        │
        ├─ 调用 ./build.sh <device> debug
        ├─ 在 action_build/ 中执行 make tools/install
        ├─ 在 action_build/ 中执行 make toolchain/install
        ├─ 在 action_build/ 中执行 make target/compile
        ├─ 在 action_build/ 中执行 make package/.../download
        ├─ 在 action_build/ 中执行 make package/.../compile
        └─ 收集 .ipk → ipk_artifacts/<device>/
```

这个路径复用了现有源码准备、`update.sh` 定制和 `.config` 合并流程，但把最终编译范围缩小到指定包。

### 3. 源码锁定与可追溯

每次准备构建副本时,[source-common.sh#write_source_lock](../scripts/source-common.sh) 会写入 `action_build/.source-lock`:

```
device=jdcloud_ipq60xx_immwrt
source_path=sources/imm-nss
source_url=https://github.com/VIKINGYFY/immortalwrt.git
source_branch=main
source_commit=a1b2c3d4e5f6...
locked_at_utc=2026-06-30T09:00:00Z
```

这确保每次构建都能追溯到精确的上游 commit。`update-source-locks.sh` 可批量更新 `metadata/sources/<id>.lock` 作为快照存档。

### 4. 外部依赖登记与校验

所有外部 URL(git 仓库、raw 文件、binary feed、API)必须登记在 [metadata/external-dependencies.tsv](../metadata/external-dependencies.tsv) 中。CI 与本地构建前,[validate-source.sh](../scripts/validate-source.sh) 会:

1. 扫描 `wrt_core/*.sh`、`wrt_core/modules/*.sh`、`wrt_core/compilecfg/*.ini` 中所有 `https://` URL
2. 逐一校验是否在 tsv 中登记
3. 校验目标 submodule 已在 `.gitmodules` 注册且已初始化
4. 缺失任一即 fail-fast

这避免了"脚本里加了个新 URL 但没记录"导致的供应链盲区。tsv 字段:`category | name | url | branch_or_ref | owner | file | purpose`,覆盖 upstream-source / feed / package-feed / package / raw / binary-feed / release-asset / api 等类别。

### 5. 网络重试封装

[network.sh](../wrt_core/modules/network.sh) 提供 `git_retry` / `curl_retry` / `wget_retry`,统一处理网络不稳定:

- 指数退避:5s → 10s → 20s → 40s → 80s
- 最大重试 5 次
- `git_retry clone` 失败时自动清理半成品目录

### 6. 幂等性

多数定制函数遵循"先检查再改"的幂等模式:

```bash
_docker_stack_patch_dockerman_backend_helpers() {
    local dockerman_init="$1"
    grep -Fq 'dockerman_use_iptables() {' "$dockerman_init" && return 0
    # 否则打补丁
    ...
}
```

这使得**重复运行 update.sh 安全**,便于调试。注意:`WRT_LOCAL_SOURCE=1` 模式下,`clone_repo` 与 `reset_feeds_conf` 会跳过 git 操作(因为副本已由 prepare-source.sh 准备好)。

### 7. Sparse Checkout 优化

[install_custom_feed](../wrt_core/modules/packages.sh) 从 4 个第三方仓库拉取指定包时,使用 git sparse-checkout **只下载需要的目录**:

```bash
clone_args=(clone --depth 1 --filter=blob:none --sparse)
git_retry "${clone_args[@]}" "$repo_url" "$tmp_dir"
git_retry -C "$tmp_dir" sparse-checkout set "${packages[@]}"
```

### 8. 错误屏障

[update.sh](../wrt_core/update.sh) 顶部设置:

```bash
set -e
set -o errexit
set -o errtrace
trap 'error_handler' ERR
```

任何步骤失败立即停止,避免错误传播。

## 数据流

### 完整构建数据流(submodule 模式,默认)

```
用户执行 ./build.sh jdcloud_ipq60xx_immwrt
        │
        ▼
build.sh 读取 compilecfg/jdcloud_ipq60xx_immwrt.ini
        │  → BUILD_DIR=imm-nss (即 source-id)
        │  → REPO_URL / REPO_BRANCH (与 .gitmodules 一致,用于回退模式)
        │
        ▼
scripts/prepare-source.sh jdcloud_ipq60xx_immwrt
        │  → resolve_device_source → sources/imm-nss
        │  → 校验 sources/imm-nss/.git 存在(submodule 已初始化)
        │  → rsync sources/imm-nss/ → action_build/ (排除 .git/.ccache/...)
        │  → write_source_lock → action_build/.source-lock
        │
        ▼
build.sh 设置 BUILD_DIR=action_build, export WRT_LOCAL_SOURCE=1
        │
        ▼
update.sh 执行(对 action_build/ 目录)
        │  (clone_repo / reset_feeds_conf 检测 WRT_LOCAL_SOURCE=1 跳过 git 操作)
        ├─ clean_up            → 清理 .config / tmp / feeds
        ├─ update_feeds        → 注入 bandix feed,./scripts/feeds update -a
        ├─ remove_unwanted_packages → 删除 passwall/ssr-plus 等重复包
        ├─ install_custom_feed → sparse clone 4 个第三方仓库到 custom_feed/
        ├─ fix_netfilter_kmod_clash → 修补 netfilter.mk (Linux 6.12 兼容)
        ├─ docker_stack_sync_nftables_compat → 重写 dockerd init
        ├─ ...(共约 60 步)
        └─ install_feeds       → ./scripts/feeds install -a -f
        │
        ▼
build.sh apply_config
        │  cp deconfig/jdcloud_ipq60xx_immwrt.config → action_build/.config
        │  cat nss.config >> .config (IPQ 设备)
        │  cat compile_base.config >> .config
        │  cat docker_deps.config >> .config
        │  cat proxy.config >> .config
        │
        ▼
cd action_build && make defconfig  (展开依赖、解决冲突)
        │
        ▼
make download -j24  (下载源码包,带重试)
        │
        ▼
make -j13  (编译,失败则 make -j1 V=s)
        │
        ▼
find bin/targets -name "*.bin|*.itb|*.fip|..." → cp firmware/
        │
        ▼
make clean  (释放 action_build 空间)
        │
        ▼
输出 firmware/ 目录(含 .bin / .manifest / .itb 等)
```

### CI 构建数据流

GitHub Actions 流程:

```
workflow_dispatch(选设备)
   → Checkout (submodules: false, 只拉主仓库)
   → Resolve Source (从 ini 读 BUILD_DIR → source_id)
   → Init Target Source Submodule (只初始化 sources/<source_id>, --depth 1)
   → Validate Source (校验 submodule + 外部依赖登记)
   → Export Source Commit (记录精确 commit,用于缓存 key)
   → Cache 恢复(action_build/.ccache + staging_dir,key 含 source_commit)
   → build.sh (内部走 prepare-source → update.sh → make)
   → Validate Firmware Output (校验产物数量 > 0)
   → Delete Old Cache → Upload Artifact
```

关键差异:
- CI **不**在 checkout 时拉所有 submodule(`submodules: false`),而是按设备只初始化一个
- 缓存 key 基于 `source_commit`(精确到源码 commit)+ 脚本/配置哈希 + 日期,确保源码变更即新缓存
- 缓存 path 为 `action_build/.ccache` 与 `action_build/staging_dir`

### 独立 IPK 发布数据流

```
workflow_dispatch(选设备 + package_targets)
   → Checkout
   → Resolve Source
   → Init Target Source Submodule
   → Validate Source
   → Restore Cache
   → scripts/build-ipk.sh
       → ./build.sh <device> debug
       → make tools/install
       → make toolchain/install
       → make target/compile
       → make package/.../download
       → make package/.../compile
       → copy .ipk -> ipk_artifacts/<device>/
   → Validate IPK Output
   → Release GitHub assets
```

```
  ├── scripts/build-ipk.sh (独立包构建入口,内部复用 build.sh debug)

  │     └── scripts/source-common.sh (resolve_device_source, write_source_lock, ...)
  ├── scripts/validate-source.sh (CI 中调用)
  │     └── scripts/source-common.sh
  └── wrt_core/update.sh
        ├── source general.sh    → clone_repo, clean_up, reset_feeds_conf (支持 WRT_LOCAL_SOURCE)
        ├── source network.sh    → git_retry, curl_retry, wget_retry (基础设施)
        ├── source feeds.sh      → update_feeds, install_feeds
        ├── source packages.sh   → remove_unwanted_packages, install_custom_feed, update_*
        ├── source system.sh     → fix_*, update_*, set_*, change_*

```

## 扩展点

### 添加新设备

1. 确定上游 source-id(已有 6 个,见 [设备适配](设备适配.md));若需新上游,先在 `.gitmodules` 添加 submodule
2. 创建 `compilecfg/<new_device>.ini`(`BUILD_DIR` 填对应 source-id)
3. 创建 `deconfig/<new_device>.config`
4. 在 `.github/workflows/build_wrt.yml` 的 `options` 添加设备名
5. 若引入新的外部 URL,必须在 `metadata/external-dependencies.tsv` 登记

### 添加新定制步骤

1. 在合适的 module(如 `system.sh`)中编写函数,遵循幂等模式
2. 在 `update.sh` 的 `main()` 中按依赖顺序插入调用
3. 若需要补丁文件,放入 `patches/`

### 修改上游源码

1. 进入对应 `sources/<source-id>` submodule
2. 提交并推送到可访问的 fork/镜像(CI 需能拉取)
3. 回主仓库提交 submodule 指针:`git add sources/<source-id> && git commit`
4. 可用 `./scripts/export-source-patches.sh <source-id>` 导出工作树差异为 patch 存档

详见 [上游源码管理](source-management.md)。

## 架构权衡

### 优点
- **源码可审计**:submodule 指针锁定上游 commit,变更走 git 流程
- **CI 稳定**:不依赖运行时 clone,源码版本固定
- **供应链透明**:外部依赖集中登记,validate-source 强制校验
- **构建副本隔离**:submodule 工作树保持干净,缓存可独立管理
- **定制可追踪**:所有修改以脚本形式存在,git 可追溯

### 代价
- **submodule 复杂度**:克隆需 `--recurse-submodules` 或按设备手动初始化
- **源码修改链路长**:本地改源码需在 submodule 内提交推送,再更新主仓库指针
- **脆弱性**:定制脚本依赖上游文件的具体格式(如 docker.sh 的 awk 匹配),上游改格式则补丁失效
- **无单元测试**:shell 脚本缺少测试覆盖,依赖实际编译验证
